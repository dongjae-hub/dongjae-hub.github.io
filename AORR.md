# AORR 상태 머신 설계

이 문서는 `dongjae-hub.github.io` 저장소를 대상으로 한 개인 프로페셔널 웹사이트 개발 업무를, GitHub Pages에서 동작하는 정적 웹사이트 기준의 실행 가능한 AORR 상태 머신으로 정리한 설계서다.

설계 원칙:

- 최종 결과물은 백엔드 없이 `HTML + CSS + JavaScript`만으로 동작해야 한다.
- 루프는 항상 `한 번에 하나의 실패 원인`만 고친다.
- 수정 범위는 가능한 한 최소 파일만 포함한다.
- 각 루프는 `Observe -> Reason -> Repeat`로 닫히며, 성공하면 다음 작은 루프로 이동한다.
- 개인 정보, 콘텐츠, 배포 권한, GitHub 설정처럼 사람이 결정해야 하는 항목은 `[사람 확인 필요]`로 표시한다.

## 1. Target

### 프로페셔널 웹사이트 개발 목표

- 개인 프로페셔널 웹사이트를 정적 사이트로 구현한다.
- 첫인상은 깔끔하고 신뢰감 있게 보이도록 한다.
- 모바일과 데스크톱 모두에서 읽기 쉽고 조작하기 쉬워야 한다.
- 상단 내비게이션에서 `Games` 탭을 통해 게임 페이지로 이동할 수 있어야 한다.

### GitHub Pages 배포 목표

- GitHub Pages에서 직접 서비스 가능한 정적 결과물을 만든다.
- 배포 산출물은 공개 가능한 정적 파일만 포함해야 한다.
- GitHub Pages 호환성을 유지해야 한다.

### 입력 자료

- 현재 저장소의 기존 파일 구조
- 현재 구현된 HTML/CSS/JS
- 사용자가 제공하는 개인 소개, 경력, 프로젝트, 링크, 연락처
- `Step 1` 분석 결과에서 확인된 작업 범위
- 필요 시 `[사람 확인 필요]`로 남겨야 하는 콘텐츠 판단

### 필수 페이지와 섹션

- `index.html` 기반 홈 페이지
- 프로페셔널 소개 영역
- 프로젝트 또는 작업 영역
- 핵심 가치 또는 소개 섹션
- 연락처 또는 외부 링크 섹션
- 상단 내비게이션
- `Games` 페이지 또는 게임 섹션

### Games 탭 및 지렁이 게임 요구사항

- 상단 내비게이션에 `Games` 탭이 있어야 한다.
- 지렁이 게임은 정적 JavaScript만으로 동작해야 한다.
- 키보드로 조작 가능해야 한다.
- 모바일 터치로 조작 가능해야 한다.
- 게임 상태, 점수, 시작/재시작 같은 피드백이 보여야 한다.
- 접근 가능한 상태 문구와 조작 안내가 있어야 한다.
- `Step 1`에서 확인된 게임 요구사항은 현재 분석 기준으로 `Loop Snake` 성격의 지렁이 게임이다.
- 추가적인 게임 룰이나 연출 요구가 따로 있었다면 `[사람 확인 필요]`로 재확인해야 한다.

### 데스크톱 및 모바일 완료 기준

- 320px, 768px, 1440px에서 가로 스크롤이 없어야 한다.
- 내비게이션과 주요 버튼은 터치하기 쉬운 크기여야 한다.
- 텍스트는 잘리지 않고 읽혀야 한다.
- 게임 보드와 조작 버튼이 화면 안에 안정적으로 들어와야 한다.
- 키보드와 터치 모두에서 게임 입력이 기대대로 작동해야 한다.
- GitHub Pages 공개 경로에서 정적 자산이 정상 참조되어야 한다.

## 2. Act

### 한 번의 개발 루프에서 수행할 최소 작업

1. 하나의 실패 원인만 선택한다.
2. 해당 원인에 직접 연결된 최소 파일만 수정한다.
3. 관련된 최소 테스트 또는 검증만 실행한다.
4. 그 실패가 해결됐는지 확인한다.
5. 기존에 통과한 기능에 회귀가 없는지 최소 범위로 다시 확인한다.

### 수정 가능한 파일 범위

- 루프와 직접 관련된 HTML 파일
- 루프와 직접 관련된 CSS 파일
- 루프와 직접 관련된 JavaScript 파일
- 테스트 파일
- 빌드 스크립트

### 생성할 수 있는 파일

- `index.html`
- `styles.css`
- `script.js`
- 게임용 JavaScript 파일 또는 `script.js` 내부의 게임 코드
- 테스트 파일
- GitHub Pages용 정적 산출물 설정 파일

### 실행 가능한 로컬 검증 명령어

- `npm test`
- `npm run build`
- 브라우저 수동 확인
- 로컬 정적 서버 확인
- 필요 시 `git diff --check`

## 3. Observe

### 파일 생성 여부

- 필요한 루트 파일이 존재하는지 확인한다.
- 산출물이 GitHub Pages 공개 구조와 맞는지 확인한다.

### HTML, CSS, JavaScript 오류

- HTML 구조가 깨지지 않았는지 확인한다.
- CSS가 오버플로우나 잘림을 만들지 않는지 확인한다.
- JavaScript 구문 오류와 런타임 오류를 확인한다.

### 로컬 웹서버 응답

- 홈 페이지가 응답하는지 확인한다.
- `Games` 진입 경로가 응답하는지 확인한다.
- 정적 자산이 정상 로드되는지 확인한다.

### 브라우저 콘솔 오류

- 콘솔에 오류가 없는지 확인한다.
- 경고가 기능 저하를 의미하는지 판단한다.

### 데스크톱 및 모바일 화면

- 데스크톱 폭에서 레이아웃이 자연스러운지 확인한다.
- 모바일 폭에서 내비게이션과 게임 UI가 무너지지 않는지 확인한다.

### 키보드 및 터치 게임 조작

- 키보드 입력이 방향 전환으로 이어지는지 확인한다.
- 터치 입력이 방향 전환으로 이어지는지 확인한다.
- 잘못된 역방향 입력이 규칙상 차단되는지 확인한다.

### GitHub Pages 호환성

- 정적 파일만으로 동작하는지 확인한다.
- 상대 경로가 GitHub Pages 경로와 맞는지 확인한다.
- 빌드 산출물이 공개 저장소 구조에 맞는지 확인한다.

## 4. Reason

실패 원인은 아래 하나로만 분류한다.

- `HTML_STRUCTURE`: 마크업 구조, 시맨틱 태그, 링크, 페이지 배치 문제
- `CSS_RESPONSIVE`: 반응형, 오버플로우, 터치 타깃, 레이아웃 문제
- `JAVASCRIPT`: 문법 오류, 모듈 로드 실패, 일반 런타임 오류
- `GAME_LOGIC`: 점수, 충돌, 성장, 방향 검증 같은 게임 규칙 문제
- `GAME_CONTROL`: 키보드, 스와이프, 버튼 입력 변환 문제
- `CONTENT`: 소개 문구, 경력, 프로젝트, 연락처 등 내용 문제
- `TEST`: 테스트 기대값, 테스트 범위, 테스트 깨짐
- `ENVIRONMENT`: 로컬 도구, Git, Node, 브라우저, 파일 시스템 문제
- `GITHUB_PERMISSION`: GitHub 인증, 권한, 토큰, 저장소 접근 문제
- `DEPLOYMENT`: GitHub Pages 배포, 공개 경로, 배포 산출물 문제
- `UNKNOWN`: 위 분류로도 확정할 수 없는 경우

분류 기준:

- HTML이 깨졌거나 필요한 요소가 없으면 `HTML_STRUCTURE`
- 모바일에서만 문제가 나면 우선 `CSS_RESPONSIVE`
- 키보드/터치 입력이 다르게 동작하면 `GAME_CONTROL`
- 게임이 규칙과 다르게 진행되면 `GAME_LOGIC`
- 텍스트, 이름, 프로젝트 설명이 불명확하거나 누락되면 `CONTENT`
- 테스트 기대값이 실제 요구와 불일치하면 `TEST`
- 도구나 실행 환경 자체의 문제면 `ENVIRONMENT`
- 토큰, 인증, 저장소 접근이 막히면 `GITHUB_PERMISSION`
- 배포된 사이트가 다르게 보이거나 열리지 않으면 `DEPLOYMENT`

## 5. Repeat

- 한 번에 하나의 실패 원인만 수정한다.
- 관련된 최소 파일만 변경한다.
- 수정 후 동일한 Verifier를 다시 실행한다.
- 이미 통과한 영역에 대해 회귀 테스트를 함께 확인한다.
- 같은 실패 fingerprint가 다시 나오면 같은 루프를 반복하지 말고 원인 재분류를 먼저 한다.

권장 반복 순서:

1. 실패 분류
2. 최소 수정
3. 동일 Verifier 재실행
4. 회귀 확인
5. 통과 시 다음 루프로 이동

## 6. Stop

다음 중 하나면 해당 루프를 멈춘다.

- 전체 테스트가 통과한 경우
- 최대 Retry에 도달한 경우
- 동일한 오류 fingerprint가 2회 반복된 경우
- 개인정보나 콘텐츠 확인이 필요한 경우
- GitHub 인증 또는 배포 권한 문제가 발생한 경우

## 7. Human-in-the-loop

다음 상황에서는 사람 확인이 필요하다.

- 이름, 소개, 경력, 프로젝트 등 개인 콘텐츠가 불명확한 경우
- 기존 콘텐츠 삭제가 필요한 경우
- 외부 분석 도구나 외부 서비스를 추가해야 하는 경우
- GitHub 저장소 설정을 변경해야 하는 경우
- 요구사항이 충돌하는 경우
- 게임 규칙, 분위기, 난이도, 화면 구성에 대한 선호가 여러 해석으로 갈리는 경우

## 상태 머신

### 상태 정의

- `READY`: 루프 시작 전, 입력과 범위를 정리한 상태
- `ACTING`: 최소 수정 작업을 수행하는 상태
- `VERIFYING`: 동일 Verifier로 결과를 확인하는 상태
- `RETRYING`: 실패 원인을 하나로 좁혀 다시 수정하는 상태
- `PASSED`: 해당 루프가 성공한 상태
- `DEPLOY_READY`: 배포 전 최종 검증이 끝난 상태
- `DEPLOYING`: GitHub Pages 배포를 수행하는 상태
- `DEPLOYED`: 배포와 스모크 테스트까지 끝난 상태
- `BLOCKED`: 환경이나 권한 때문에 더 진행할 수 없는 상태
- `HITL_REQUIRED`: 사람 확인이 필요한 상태

### 전이 규칙

| 현재 상태 | 입력/조건 | 다음 상태 | 의미 |
|---|---|---|---|
| `READY` | 범위와 실패 원인이 명확함 | `ACTING` | 최소 수정 시작 |
| `READY` | 콘텐츠나 결정이 불명확함 | `HITL_REQUIRED` | 사람 확인 대기 |
| `ACTING` | 수정 완료 | `VERIFYING` | 검증으로 이동 |
| `VERIFYING` | 통과 | `PASSED` | 루프 성공 |
| `VERIFYING` | 하나의 실패 원인만 남음 | `RETRYING` | 최소 수정 재시도 |
| `VERIFYING` | 인증/환경/배포 불가 | `BLOCKED` | 외부 조건 차단 |
| `RETRYING` | 수정 완료 | `VERIFYING` | 동일 Verifier 재실행 |
| `RETRYING` | 반복 실패가 2회 이상 | `BLOCKED` | 더 이상의 자동 반복 중지 |
| `PASSED` | 다음 루프 존재 | `READY` | 다음 작업 시작 |
| `PASSED` | 배포 전 루프 완료 | `DEPLOY_READY` | 배포 준비 완료 |
| `DEPLOY_READY` | 배포 승인 및 권한 정상 | `DEPLOYING` | 배포 시작 |
| `DEPLOYING` | 배포와 스모크 테스트 성공 | `DEPLOYED` | 공개 상태 완료 |
| `DEPLOYING` | 배포 실패 | `RETRYING` 또는 `BLOCKED` | 원인에 따라 재시도 또는 차단 |
| `BLOCKED` | 사람 확인 또는 환경 복구 | `READY` | 다시 진행 가능 |
| `HITL_REQUIRED` | 사람 응답 수신 | `READY` | 재개 가능 |

## 단계별 작은 루프 표

| 단계 | 상태 | 입력 | Act | Observe | 출력 | 테스트 기준 | 다음 상태 |
|---|---|---|---|---|---|---|---|
| 저장소 및 기존 파일 확인 | `READY` | 현재 저장소, 루트 파일, 기존 구조 | 파일 목록과 주요 문서를 확인한다 | 무엇이 이미 존재하는지, 누락된 파일이 무엇인지 확인한다 | 작업 범위 초안 | 루트 구조와 현재 스택 파악 완료 | `PASSED` 또는 `HITL_REQUIRED` |
| 정적 사이트 기본 구조 | `READY` -> `ACTING` | 현재 HTML 구조, 정적 사이트 목표 | `index.html`, 기본 레이아웃, 메타 태그, 정적 경로를 정리한다 | 기본 페이지가 열리고 링크가 살아 있는지 본다 | 홈 기본 골격 | HTML 오류 없음, 홈 열림, 페이지 전환 가능 | `PASSED` |
| 프로페셔널 콘텐츠 영역 | `READY` -> `ACTING` | 소개 문구, 프로젝트, 링크, 연락처 | 프로필과 작업 소개를 배치한다 | 내용이 전문적으로 읽히는지 확인한다 | 홈 콘텐츠 영역 | 사실 정보가 정확해야 함 | `HITL_REQUIRED` 또는 `PASSED` |
| 반응형 내비게이션 | `READY` -> `ACTING` | 모바일/데스크톱 폭, 내비 구조 | 상단 내비게이션을 반응형으로 맞춘다 | 320px에서 링크가 눌리는지 본다 | 모바일 대응 내비 | 가로 스크롤 없음, 탭 가능 | `PASSED` |
| Games 탭 | `READY` -> `ACTING` | 홈/게임 간 진입 흐름 | `Games` 진입점을 추가하거나 정리한다 | 홈에서 게임으로 이동 가능한지 확인한다 | 게임 진입 경로 | 링크 정상, 현재 페이지 표시 가능 | `PASSED` |
| 지렁이 게임 핵심 로직 | `READY` -> `ACTING` | 게임 규칙, 점수, 충돌, 성장 | 순수 게임 로직을 구현한다 | 움직임, 성장, 종료가 규칙대로인지 본다 | 게임 상태 머신 | deterministic random 기준 테스트 통과 | `PASSED` |
| 키보드 조작 | `READY` -> `ACTING` | 방향키, WASD | 키 입력을 방향 명령으로 변환한다 | 입력이 즉시 방향 전환으로 이어지는지 본다 | 키보드 조작 가능 | 역방향 차단, 입력 반응 정상 | `PASSED` |
| 모바일 터치 조작 | `READY` -> `ACTING` | 터치 제스처, 방향 버튼 | 스와이프와 터치 버튼을 방향 명령에 연결한다 | 모바일에서 조작 가능한지 본다 | 터치 조작 가능 | 스와이프 임계값, 버튼 동작, 반응성 | `PASSED` |
| 게임 UI 및 점수 | `READY` -> `ACTING` | 점수 UI, 상태 문구, 시작/재시작 | 점수, 상태, 안내, 시작 제어를 정리한다 | 사용자가 현재 상태를 이해하는지 본다 | 게임 UI 완성 | 접근 가능한 상태 표시, 점수 갱신 | `PASSED` |
| 접근성과 반응형 검증 | `READY` -> `VERIFYING` | 화면 폭, 콘솔, 접근성 속성 | 브라우저와 테스트로 검증한다 | 콘솔 오류, 오버플로우, 포커스 가능성 확인 | 검증 리포트 | 320/768/1440에서 안정적 | `PASSED`, `RETRYING`, `BLOCKED` |
| GitHub Pages 호환성 검증 | `READY` -> `VERIFYING` | 정적 산출물, 상대 경로, 공개 자산 | Pages 경로와 정적 산출물을 확인한다 | 공개 경로에서 파일이 로드되는지 확인한다 | 배포 가능성 판단 | 백엔드 없이 정상 실행 | `DEPLOY_READY` 또는 `BLOCKED` |
| 배포 | `DEPLOY_READY` -> `DEPLOYING` | 승인된 정적 산출물, GitHub 권한 | GitHub Pages에 배포한다 | 공개 URL 응답과 자산 로드 확인 | 배포 완료 결과 | 홈, Games, CSS, JS 응답 200 | `DEPLOYED` |

## 루프 운영 메모

- 루트에 필요한 파일은 최소한 `index.html`, `styles.css`, `script.js`가 되어야 한다.
- 게임이 분리 파일을 쓰더라도 최종 배포 구조는 GitHub Pages에서 바로 동작해야 한다.
- 현재 저장소의 기존 구조가 `assets/`와 `games/`로 나뉘어 있다면, 두 가지 중 하나를 선택해야 한다.
  - 루트 파일로 평탄화한다.
  - 빌드 단계에서 루트 산출물을 생성한다.
- 이 선택은 구조 변경 폭이 크므로, 실제 적용 전 `[사람 확인 필요]`다.

## Step 1에서 확인된 게임 추가 기능 반영

Step 1 분석 기준으로 별도 추가 게임 기능 요구는 확인되지 않았고, 게임 요구는 `지렁이 게임(Loop Snake)` 중심으로 정리하는 것이 안전하다.

만약 이후에 추가 요구가 발견되면 다음 순서로 반영한다.

1. 입력 방식 추가
2. 게임 규칙 변경
3. UI 연출 추가
4. 배포 기준 재검토

## 권장 초기 실행 순서

1. 저장소 및 기존 파일 확인
2. 정적 사이트 기본 구조
3. 프로페셔널 콘텐츠 영역
4. 반응형 내비게이션
5. Games 탭
6. 지렁이 게임 핵심 로직
7. 키보드 조작
8. 모바일 터치 조작
9. 게임 UI 및 점수
10. 접근성과 반응형 검증
11. GitHub Pages 호환성 검증
12. 배포

## 최종 판정 규칙

- 어떤 루프든 `Content`, `GitHub Permission`, `Deployment`, `Environment`, `Human-in-the-loop` 사유가 있으면 자동 진행을 멈춘다.
- 기능이 통과해도 회귀가 있으면 다음 루프로 넘어가지 않는다.
- 배포는 검증 완료 후에만 진행한다.

## Self-Correcting TDD Loop

이 섹션은 이 저장소에서 실제로 존재하거나 현재 환경에서 실행 가능한 검증기만 기준으로 구성한 Verifier 중심 루프다.

### 실제 확인된 검증 도구

| 도구 | 확인 결과 | 역할 |
|---|---|---|
| `npm test` | 존재함 (`package.json`) | 정적/단위 TDD 검증의 기본 Verifier |
| `npm run build` | 존재함 (`package.json`) | GitHub Pages용 정적 산출물 검증 |
| `node --test` | `npm test`가 호출 | HTML/JS/게임 로직 회귀 검사 |
| `node scripts/build.mjs` | `npm run build`가 호출 | 배포 산출물 생성 검증 |
| `git diff --check` | 사용 가능 | 공백/패치 손상/커밋 전위성 검증 |
| `claude` CLI | 설치됨, 버전 `2.1.208 (Claude Code)` | 독립 Verifier 후보 |
| Claude Code 모델 | `--model sonnet`가 `claude-sonnet-5`로 해석됨 | 모델 기반 2차 Verifier |
| `python3 -m http.server` | 이 환경에서는 Python 미설치로 사용 불가 | 있으면 사용 가능한 로컬 정적 서버 |
| Node 기반 임시 정적 서버 | 사용 가능 (`node` 설치됨) | Python이 없을 때의 로컬 HTTP Verifier |

Claude Code 확인 메모:

- `claude --help`에서 `--model` 인자가 실제로 지원됨을 확인했다.
- 라이브 호출 메타데이터에서 `claude-sonnet-5`가 사용된 것을 확인했다.
- 따라서 이 프로젝트의 독립 Verifier로 Claude Code CLI를 쓸 때의 실제 모델명은 `claude-sonnet-5`로 기록한다.

### Verifier 선택 원칙

1. 먼저 가장 좁은 Verifier를 쓴다.
2. 실패가 코드보다 문서/구조/경로인지 먼저 분리한다.
3. 실패가 브라우저에서만 보이면 브라우저 Verifier로 확인한다.
4. 실패가 빌드 산출물에서만 보이면 `npm run build`를 우선한다.
5. 실패가 입력/상호작용이면 게임 Verifier를 따로 분리한다.
6. 실패가 환경/권한이면 코드 수정이 아니라 상태 전환으로 처리한다.

### 권장 Verifier 순서

| 순서 | Verifier | 확인 대상 | 통과 의미 |
|---|---|---|---|
| 1 | 파일/경로 검사 | `index.html`, CSS/JS 연결, 대소문자, 절대 경로 | 루트 구조와 참조가 GitHub Pages 친화적 |
| 2 | HTML 검사 | 문서 구조, `title`, `meta viewport`, nav, Games, alt | 기본 마크업이 유효 |
| 3 | JS 정적 검사 | 모듈 로드, null 참조 가능성, 문법 위험 | 초기 로드 오류 가능성 낮음 |
| 4 | `npm test` | 기존 테스트 전체 | TDD 기준의 기본 회귀 |
| 5 | `npm run build` | 빌드 산출물 | Pages 배포 가능성 확인 |
| 6 | 로컬 HTTP 서버 | 홈/게임/CSS/JS 응답 | 상대 경로와 정적 제공 확인 |
| 7 | 브라우저 Verifier | 375px, 768px, 1440px | 레이아웃/콘솔/상호작용 확인 |
| 8 | Claude Code 독립 Verifier | 구조, 리스크, 회귀 해석 | 2차 리뷰와 실패 분류 보강 |

### Self-Correcting 상태 흐름

| 상태 | 트리거 | 다음 상태 | 설명 |
|---|---|---|---|
| `READY` | 검증기와 범위가 확정됨 | `VERIFYING` | 먼저 확인하고 시작 |
| `VERIFYING` | Verifier 통과 | `PASSED` | 해당 범위 완료 |
| `VERIFYING` | 하나의 실패 원인 발견 | `RETRYING` | 최소 수정으로 복귀 |
| `RETRYING` | 최소 수정 적용 | `VERIFYING` | 동일 Verifier 재실행 |
| `RETRYING` | 같은 fingerprint 2회 반복 | `BLOCKED` | 자동 반복 중지 |
| `RETRYING` | 콘텐츠/권한/환경 확인 필요 | `HITL_REQUIRED` | 사람 확인 요청 |
| `PASSED` | 다음 범위 존재 | `READY` | 다음 루프로 이동 |

### 실패 로그 수집 규격

실패가 나면 아래 항목을 반드시 남긴다.

| 항목 | 내용 |
|---|---|
| 실행 명령어 | 실제 실행한 Verifier 명령 |
| exit code | 종료 코드 |
| 실패한 검증 항목 | 어떤 체크가 깨졌는지 |
| 핵심 오류 메시지 | 에러의 본문과 요지 |
| 관련 파일과 라인 | 가능하면 파일 경로와 라인 |
| 브라우저 콘솔 메시지 | 콘솔/경고/런타임 오류 |
| 오류 fingerprint | 재현 가능한 핵심 패턴 |
| 분류 | `HTML_STRUCTURE` 등 하나만 선택 |
| 가설 | 왜 이 실패가 생겼는지 |
| 변경 파일 | Retry에서 수정한 최소 파일 |
| 재실행 Verifier | 수정 후 다시 돌릴 동일 검증기 |

### 기본 TDD 루프

1. `READY`
   - 입력: 저장소 구조, 현재 AORR, 존재하는 검증 도구
   - 행동: 이번 루프의 실패 원인 하나와 Verifier 하나를 고른다

2. `VERIFYING`
   - 입력: 선택한 Verifier
   - 행동: 현재 상태를 깨는지 확인한다
   - 출력: 통과 또는 실패 로그

3. `REASON`
   - 입력: 실패 로그
   - 행동: 실패를 단일 원인으로 분류한다
   - 출력: 하나의 분류 라벨

4. `RETRYING`
   - 입력: 분류 라벨과 최소 가설
   - 행동: 관련 파일만 최소 수정한다
   - 출력: 수정된 파일 집합

5. `VERIFYING`
   - 입력: 동일 Verifier
   - 행동: 같은 조건으로 다시 확인한다
   - 출력: 통과 또는 동일 fingerprint 재발

6. `PASSED`
   - 입력: 모든 필수 Verifier 통과
   - 행동: 회귀 범위만 간단히 확인하고 다음 루프로 이동한다
   - 출력: 다음 작업 준비

### 루프별 최소 수정 원칙

- `HTML_STRUCTURE`는 HTML만, 필요한 경우 CSS 연결만 고친다.
- `CSS_RESPONSIVE`는 CSS만 고친다.
- `JAVASCRIPT`는 JS 로드/문법/DOM 참조만 고친다.
- `GAME_LOGIC`는 게임 코어만 고친다.
- `GAME_CONTROL`는 입력 변환만 고친다.
- `CONTENT`는 텍스트와 정보만 고친다.
- `TEST`는 테스트 기대값이 실제 요구와 어긋난 경우에만 고친다.
- `ENVIRONMENT`, `GITHUB_PERMISSION`, `DEPLOYMENT`는 코드가 아니라 상태/권한/실행환경으로 처리한다.

### 권장 Verifier-문제 매핑

| 실패 증상 | 우선 Verifier | 주 분류 |
|---|---|---|
| `index.html`가 없거나 링크가 깨짐 | 파일/경로 검사, HTML 검사 | `HTML_STRUCTURE` |
| 모바일에서 가로 스크롤 발생 | 브라우저 Verifier | `CSS_RESPONSIVE` |
| 페이지 로드 시 JS 예외 | JS 정적 검사, 브라우저 콘솔 | `JAVASCRIPT` |
| 점수가 안 오르거나 충돌 판정이 틀림 | `npm test` + 게임 Verifier | `GAME_LOGIC` |
| 키보드/터치 입력이 다르게 동작 | 브라우저 Verifier + 게임 입력 테스트 | `GAME_CONTROL` |
| 내용이 비어 있거나 사실과 다름 | 문서/콘텐츠 확인 | `CONTENT` |
| 테스트가 요구와 맞지 않음 | `npm test` | `TEST` |
| 명령이 없거나 도구가 막힘 | 도구 존재성 검사 | `ENVIRONMENT` |
| 토큰/권한/로그인이 필요함 | 인증 상태 확인 | `GITHUB_PERMISSION` |
| Pages에서 열리지만 자산이 안 뜸 | 로컬 HTTP 서버 + Pages 경로 검사 | `DEPLOYMENT` |

### 게임 전용 Verifier 순서

1. 시작
2. 일시정지
3. 다시 시작
4. 점수 증가
5. 음식 생성
6. 벽 충돌
7. 자기 몸 충돌

11. 터치/스와이프

### 브라우저 검증 viewport

가능한 브라우저 Verifier가 있을 때는 아래 폭을 우선 확인한다.

- 모바일: 약 `375px`
- 태블릿: 약 `768px`
- 데스크톱: 약 `1440px`

### 종료 조건 보강

- 같은 fingerprint가 2회 반복되면 `BLOCKED`로 전환한다.
- 사람이 결정해야 하는 콘텐츠가 나오면 `HITL_REQUIRED`로 전환한다.
- 배포 권한이나 GitHub 인증 문제가 나오면 코드 수정을 멈춘다.
- 이미 통과한 기능을 깨는 수정이 필요해지면 루프를 재분해한다.
